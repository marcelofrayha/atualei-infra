# Observability

This MVP uses Spring Boot Actuator, Micrometer, Prometheus, Grafana, OpenTelemetry traces, Loki, Alloy and structured JSON logs.

## Stack components

- `crawler`: emits metrics, traces and JSON logs.
- `prometheus`: scrapes crawler, MVP, Loki and Alloy metrics.
- `tempo`: stores traces received through OTLP.
- `loki`: stores centralized container logs.
- `alloy`: reads Docker stdout/stderr and pushes logs to Loki.
- `grafana`: provisions dashboards and data sources for Prometheus, Tempo and Loki.

## Local stack

Start everything:

```bash
docker compose up --build
```

Useful URLs:

- App health: `http://localhost:8080/actuator/health`
- Prometheus metrics: `http://localhost:8080/actuator/prometheus`
- Prometheus UI: `http://localhost:9090`
- Grafana: `http://localhost:3000` (`admin` / `admin` by default)
- Tempo API: `http://localhost:3200`
- Loki API: `http://localhost:3100`
- Alloy UI: `http://localhost:12345`

## Log levels

- `ERROR`: unexpected failures that need engineering attention.
- `WARN`: rejected API requests, expected crawler failures, unavailable external source or invalid operational input.
- `INFO`: lifecycle events and business milestones, such as onboarding start/end, check-update start/end and created candidate versions.
- `DEBUG`: high-volume request completion and diagnostic details that should stay off by default.
- `TRACE`: reserved for local deep parser/debug sessions.

Logs are JSON and include stable correlation fields when available: `traceId`, `spanId`, `requestId`, `operation`, `runId`, `lawId`, `lawSlug`, `sessionId`, `anonymousId`.

## Product analytics

The backend exposes a lightweight product event endpoint:

```http
POST /api/v1/analytics/events
Content-Type: application/json

{
  "eventType": "PAGE_VIEW",
  "anonymousId": "browser-generated-id",
  "sessionId": "browser-session-id",
  "route": "/laws/57b93ed1-a97c-4196-aa28-50643a18a147",
  "action": "open-law",
  "metadata": {
    "source": "navbar"
  }
}
```

Supported event types: `SESSION_START`, `SESSION_END`, `RETURN`, `PAGE_VIEW`, `ACTION`, `SEARCH`, `LAW_OPEN`, `UNIT_OPEN`, `DOWNLOAD`, `ERROR`.

The service hashes user/session/anonymous IDs before storing them. The Grafana dashboard uses grouped routes so dynamic UUIDs do not explode metric cardinality.

## Grafana dashboards

The local Docker stack provisions Grafana automatically with:

- Data source `Prometheus`, backed by `http://prometheus:9090`.
- Data source `Tempo`, backed by `http://tempo:3200`.
- Data source `Loki`, backed by `http://loki:3100`.
- Folder `Atualei`.
- Dashboard `Atualei Observability`.

Open Grafana at `http://localhost:3000` with `admin` / `admin`, then use:

1. Open `Dashboards`.
2. Open folder `Atualei`.
3. Open `Atualei Observability`.
4. Set the time range to `Last 15 minutes`.
5. Click the refresh icon after running a functional test.

Useful panels:

- `API Throughput`: request rate grouped by API area and status family.
- `API Latency p95`: p95 request duration grouped by API area.
- `Endpoint Usage 1h`: per-endpoint request count over the last hour.
- `Product Events 1h`: analytics events generated by `/api/v1/analytics/events`.
- `Active Product Sessions`: current active session gauge.
- `Top Product Routes 24h`: grouped product routes from analytics events.
- `Crawler Runs 24h`: crawler/check-update run outcomes.
- `Crawler Duration p95`: p95 crawler run duration.
- `Parsed Units 24h`: count of parsed legal units by unit type.
- `Observability Targets`: health of Prometheus targets for the crawler, MVP, Loki, Alloy and Prometheus.
- `Loki Ingestion Rate`: log lines received by Loki per second.
- `Recent App Logs`: recent application log lines from Loki.

Prometheus scrapes every 15 seconds, so metrics may take one scrape interval to appear.

## Functional observability test

Use this when you want traffic that is more realistic than a health check, but still intentionally small. It is not a load test. It caps concurrency at 5 users and exercises public reads, admin reads and product analytics events.

Run with defaults:

```bash
scripts/observability-functional-test.sh
```

Run with explicit settings:

```bash
USERS=5 ITERATIONS=10 BASE_URL=http://localhost:8080 ADMIN_API_KEY=dev-secret \
  scripts/observability-functional-test.sh
```

By default it calls:

- `GET /api/v1/health`
- `GET /api/v1/laws`
- `GET /api/v1/laws/{lawId}`
- `GET /api/v1/laws/{lawId}/units`
- `GET /api/v1/laws/{lawId}/tree`
- `GET /api/v1/laws/{lawId}/search`
- `POST /api/v1/analytics/events`
- `GET /api/v1/admin/laws`
- `GET /api/v1/admin/laws/{lawId}/versions`
- `GET /api/v1/admin/law-versions/{versionId}/quality-report`

If you also want to exercise a safe admin write endpoint, enable Quality Report rerun for already published versions:

```bash
INCLUDE_ADMIN_RERUN=true USERS=5 ITERATIONS=3 scripts/observability-functional-test.sh
```

That calls:

```http
POST /api/v1/admin/law-versions/{versionId}/quality-report/rerun
```

The rerun endpoint recalculates and persists the quality report for an already published version without changing publication status or current-version state.

Expected output:

```text
Functional observability test finished
  totalRequests=500
  failedRequests=0

Status counts:
  200 450
  202 50
```

After running it, wait about 15 seconds and refresh Grafana.

## Finding traces

Use Grafana Explore:

1. Open `http://localhost:3000/explore`.
2. Select data source `Tempo`.
3. Use query type `Search`.
4. Set `Service Name` to `atualei-legal-crawler`.
5. Set the time range to `Last 15 minutes`.
6. Click `Run query`.
7. Open a trace and inspect spans like HTTP server handling, crawler operations or database-backed request paths.

The application emits trace IDs in logs as `traceId`, so you can correlate a log line with a Tempo trace.

## Finding metrics in Prometheus

Open `http://localhost:9090` and try:

```promql
sum by (area, status_family) (rate(atualei_api_requests_total[5m]))
```

```promql
histogram_quantile(0.95, sum by (le, area) (rate(atualei_api_request_duration_seconds_bucket[5m])))
```

```promql
sum by (event_type) (increase(atualei_product_events_total[15m]))
```

```promql
atualei_product_sessions_active
```

If a query returns no data, confirm that the crawler is up and Prometheus has scraped it:

```bash
curl http://localhost:8080/actuator/prometheus | grep atualei_api_requests
```

## Finding logs

The local stack runs Loki and Alloy. Alloy reads container stdout/stderr through the Docker socket and pushes logs to Loki. Grafana provisions the `Loki` data source automatically.

Use Grafana Explore:

1. Open `http://localhost:3000/explore`.
2. Select data source `Loki`.
3. Set the time range to `Last 15 minutes`.
4. Run this query for application logs:

```logql
{compose_service="crawler"}
```

Useful LogQL queries:

```logql
{compose_service="crawler", level="WARN"}
```

```logql
{compose_service="crawler"} | json | traceId != ""
```

```logql
{compose_service="crawler"} |= "API request rejected"
```

Successful 2xx request completion logs are emitted at `DEBUG` to avoid noisy local output. To see one log line per successful API request during the functional test, restart the crawler with:

```bash
LOGGING_LEVEL_ATUALEI=DEBUG docker compose up -d --force-recreate crawler
```

Then run:

```bash
USERS=5 ITERATIONS=3 scripts/observability-functional-test.sh
```

Filter a specific trace in Loki:

```logql
{compose_service="crawler"} | json | traceId = "<traceId>"
```

The JSON logs include `requestId`, `traceId`, `spanId`, `lawId`, `lawSlug`, `runId`, `sessionId` and `anonymousId` when those fields apply. These fields stay in the JSON payload instead of becoming Loki labels to avoid high-cardinality indexes.

Docker logs remain available as a local fallback:

```bash
docker compose logs -f crawler
```
